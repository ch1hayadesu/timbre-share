## 2. 详细风险评估

### 2.1 技术选型风险

#### R-001：开源语音克隆模型效果不稳定

| 项目 | 内容 |
|------|------|
| **风险描述** | GPT-SoVITS/CosyVoice 在中文场景的克隆效果可能不稳定，特别是对非标准口音、低质量音频的处理效果不确定 |
| **等级** | 🟡 中（优化后降级，原🔴） |
| **影响** | 核心功能体验受损，用户克隆音色相似度不达标（<70%），导致用户流失 |
| **触发条件** | 内部盲测克隆相似度低于70%；特定类型音频（嘈杂、方言）克隆效果差 |
| **优化措施** | 引入 `InferenceEngine` 抽象层 + 引擎注册路由，GPT-SoVITS效果不达标时一键切换至CosyVoice，业务代码零改动 |
| **应对策略** | ① 准备期进行多模型benchmark测试（GPT-SoVITS vs CosyVoice vs FishSpeech）<br/>② 建立模型效果评估标准（主观盲测 + 客观指标）<br/>③ 推理接口抽象化，引擎注册即插即用，故障切换分钟级<br/>④ MVP阶段限定音频质量要求（推荐上传清晰录音） |

#### R-002：GPU资源成本与可获取性

| 项目 | 内容 |
|------|------|
| **风险描述** | AI推理需要GPU实例，云GPU成本较高（¥2-5/小时），MVP阶段预算有限 |
| **等级** | 🟡 中 |
| **影响** | 项目成本超预算，或需降低推理质量/并发能力 |
| **应对策略** | ① TTS合成走ONNX CPU路径，不占用GPU资源<br/>② 深度克隆/剧本配音使用单GPU串行处理<br/>③ 选择性价比高的GPU实例（如RTX 4090云实例 ≈ ¥3/小时）<br/>④ GPU空闲自动休眠机制 |

---

### 2.2 性能瓶颈风险

#### R-003：AI推理响应时间超预期

| 项目 | 内容 |
|------|------|
| **风险描述** | 深度克隆（10-30分钟）和剧本配音（≤3分钟）的耗时可能超预期 |
| **等级** | 🟢 低（优化后降级，原🟡） |
| **影响** | 用户体验下降，用户等待过程中流失 |
| **优化措施** | 引入ONNX CPU兜底路径，TTS合成无需等待GPU，秒级返回；深度克隆/配音通过WebSocket实时推送进度 |
| **应对策略** | ① TTS合成走ONNX CPU路径，响应时间＜5s<br/>② 深度克隆/配音全异步 + WebSocket进度推送<br/>③ 前端展示预估耗时 + 实时进度百分比<br/>④ 完成时站内通知，用户无需在页面等待 |

#### R-004：并发克隆/配音任务排队

| 项目 | 内容 |
|------|------|
| **风险描述** | 多个用户同时提交深度克隆或配音任务时，单GPU串行处理导致排队 |
| **等级** | 🟢 低（优化后降级，原🟡） |
| **影响** | 高峰期用户等待时间延长 |
| **优化措施** | 4级优先级队列：TTS(P0/CPU)永不排队，即时克隆(P1)优先于深度克隆(P2)，配音(P3)最低 |
| **应对策略** | ① P0 TTS走CPU，不受GPU排队影响<br/>② 前端展示排队位置 + 预估等待时间<br/>③ 同一用户深度克隆限1个并发<br/>④ Prometheus监控队列深度，超阈值告警 |

### 2.3 安全风险

#### R-005：音色模型盗用/未授权使用

| 项目 | 内容 |
|------|------|
| **风险描述** | 分享的音色模型可能被滥用，或用他人声音克隆音色用于欺诈 |
| **等级** | 🟡 中 |
| **影响** | 法律风险、平台声誉受损 |
| **应对策略** | ① 音色模型文件使用签名URL（1小时有效期）<br/>② 下载记录留存完整审计日志<br/>③ 用户协议明确禁止滥用，违规封号<br/>④ MVP阶段不设审核，但预留审核字段 |

#### R-006：用户上传恶意文件

| 项目 | 内容 |
|------|------|
| **风险描述** | 上传的音频或剧本文件可能包含恶意代码 |
| **等级** | 🟢 低 |
| **影响** | 服务器被攻击、数据泄露 |
| **应对策略** | ① 文件类型/MIME双重校验<br/>② 上传文件存隔离目录，处理后移至正式区<br/>③ 限制文件可执行权限<br/>④ 文件大小硬限制（音频≤50MB，剧本≤5MB） |

### 2.4 扩展性风险

#### R-007：单体架构后期拆分成本

| 项目 | 内容 |
|------|------|
| **风险描述** | MVP使用单体架构，后期用户增长后需拆分为微服务，重构成本高 |
| **等级** | 🟢 低（优化后降级，原🟡） |
| **影响** | 架构重构需投入额外开发资源 |
| **优化措施** | 采用绞杀者模式：MVP阶段模块间通过 `Service(ABC)` 接口隔离，拆分时只需将对应package启动为独立进程 |
| **应对策略** | ① 所有模块基于抽象接口（Protocol）依赖，不直接耦合<br/>② 数据库按业务域划分Schema<br/>③ 关键模块（VoiceCloning / TTS / ScriptDub / Share）接口已定义<br/>④ 演进路径：V1单体→V2 AI独立→V3 微服务 |

#### R-008：音色模型存储量增长

| 项目 | 内容 |
|------|------|
| **风险描述** | 每个音色模型文件较大（50-500MB），用户量增长后存储成本快速上升 |
| **等级** | 🟢 低 |
| **影响** | 存储成本增加 |
| **应对策略** | ① MVP阶段使用MinIO自建，成本可控<br/>② 模型文件压缩/量化减少体积（FP16→INT8）<br/>③ 定期清理失败/未使用的音色模型<br/>④ 后续切换至阿里云OSS等低成本对象存储 |

---

### 2.5 第三方服务依赖风险

#### R-009：短信服务商可用性

| 项目 | 内容 |
|------|------|
| **风险描述** | 短信验证码依赖第三方短信服务商（如阿里云短信），服务中断影响用户注册登录 |
| **等级** | 🟡 中 |
| **影响** | 用户无法注册和登录 |
| **应对策略** | ① 选择有SLA保障的服务商（≥99.9%）<br/>② 准备备用短信通道（如腾讯云短信）<br/>③ MVP阶段可接受短暂不可用 |

#### R-010：开源模型许可证变更

| 项目 | 内容 |
|------|------|
| **风险描述** | GPT-SoVITS等开源模型后续变更许可证为商业使用受限 |
| **等级** | 🟢 低 |
| **影响** | 需要替换AI模型，重新微调 |
| **应对策略** | ① 优先选MIT/Apache 2.0许可证模型<br/>② 推理接口抽象化，可快速切换模型<br/>③ 关注社区动态，提前规划迁移 |
