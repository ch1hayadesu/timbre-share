## 1. 接口设计原则

### 1.1 总体原则

| 原则 | 说明 |
|------|------|
| **RESTful** | 遵循 REST 风格，资源 URL 使用名词复数 |
| **无状态** | 服务端不保存客户端状态，认证信息通过 Token 传递 |
| **面向资源** | URL 表示资源，HTTP Method 表示操作 |
| **统一响应** | 所有接口使用统一的响应格式 |
| **向后兼容** | 接口变更需版本管理，避免破坏现有客户端 |

### 1.2 URL命名规范

```
格式：/api/{version}/{resource}/{resource_id}/{sub_resource}

示例：
GET    /api/v1/voice/list             获取音色列表
POST   /api/v1/voice/upload           上传音频
GET    /api/v1/voice/detail/{id}      获取音色详情
DELETE /api/v1/voice/delete/{id}      删除音色
```

| 规范 | 规则 |
|------|------|
| 资源名 | 小写字母 + 下划线，使用复数 |
| 版本号 | `v1`、`v2` 形式，位于 `/api/` 之后 |
| 查询参数 | 小写驼峰，如 `pageSize`, `sortBy` |
| 路径参数 | 大括号包裹，如 `{voice_id}` |

### 1.3 HTTP方法语义

| 方法 | 语义 | 幂等 | 安全 |
|------|------|------|------|
| GET | 查询资源 | 是 | 是 |
| POST | 创建资源 / 提交操作 | 否 | 否 |
| PUT | 全量替换资源 | 是 | 否 |
| PATCH | 部分更新资源 | 否 | 否 |
| DELETE | 删除资源 | 是 | 否 |

## 2. 统一响应格式

### 2.1 成功响应

```json
{
    "code": 0,
    "message": "success",
    "data": { ... },
    "request_id": "req_20260705abc123"
}
```

### 2.2 失败响应

```json
{
    "code": 10001,
    "message": "手机号格式不正确",
    "data": null,
    "request_id": "req_20260705abc123"
}
```

### 2.3 分页响应

```json
{
    "code": 0,
    "message": "success",
    "data": {
        "items": [ ... ],
        "pagination": {
            "page": 1,
            "pageSize": 12,
            "total": 45,
            "totalPages": 4
        }
    },
    "request_id": "req_20260705abc123"
}
```

### 2.4 响应字段说明

| 字段 | 类型 | 说明 |
|------|------|------|
| `code` | int | 业务状态码，0 表示成功，非 0 表示错误 |
| `message` | string | 状态描述信息，成功固定为 "success" |
| `data` | object/null | 业务数据主体 |
| `request_id` | string | 请求追踪 ID，用于日志关联和排障 |

## 3. 接口版本管理

### 3.1 版本策略

| 策略 | 方案 |
|------|------|
| **位置** | URL 路径中，如 `/api/v1/voice/list` |
| **版本号** | 主版本号（major），如 v1, v2 |
| **兼容性** | 同一主版本内保持完全向后兼容 |
| **废弃周期** | 旧版本废弃后至少保留 30 天，提前公告 |
| **当前版本** | v1 |

### 3.2 版本变更触发条件

| 变更类型 | 是否需升级版本 |
|----------|---------------|
| 新增接口 | 否（同一主版本） |
| 新增可选字段 | 否 |
| 修改字段含义 | **是**（主版本+1） |
| 移除字段 | **是** |
| 修改接口路径 | **是** |
| 修改请求/响应结构 | **是** |

```
0        成功
```

## 5. 接口安全性要求

| 安全要求 | 方案 |
|----------|------|
| **认证方式** | JWT Token，Bearer 方式传递 |
| **Token有效期** | Access Token 24小时，刷新策略后续版本支持 |
| **权限校验** | 所有接口需校验 user_id 归属 |
| **传输加密** | HTTPS/TLS 1.3 强制 |
| **敏感数据** | 手机号应用层 AES-256 加密存储 |
| **限流策略** | 普通接口 60次/分钟/IP，下载接口 5次/分钟/用户 |
| **文件上传校验** | 前端+后端双重校验文件类型、大小、MIME |
| **文件安全** | 上传文件扫描，隔离目录暂存，处理后迁移 |
| **防SQL注入** | ORM参数化查询，禁止SQL拼接 |
| **防XSS** | 输入过滤，输出转义，CSP Header |
| **防CSRF** | SameSite Cookie + Referer校验 |
| **签名URL** | 音色模型/音频文件访问使用预签名URL（有效期1小时） |
| **审计日志** | 所有敏感操作（登录、分享、下载）记录完整日志 |

| 指标 | 目标值 | 说明 |
|------|--------|------|
| **API响应时间（p95）** | ≤500ms | 非AI推理接口 |
| **API响应时间（p99）** | ≤2s | 含AI推理的接口 |
| **并发量** | ≥100 QPS | MVP阶段 |
| **文件上传速度** | ≥5MB/s | 服务器端接收 |
| **搜索响应时间** | ≤500ms | 音色市场搜索 |
| **WebSocket推送延迟** | ≤1s | 克隆/配音进度推送 |

## 7. 通用规范

### 7.1 日期时间格式

- 请求/响应中使用 ISO 8601 格式：`2026-07-05T10:30:00+08:00`
- 数据库存储使用 UTC：`2026-07-05T02:30:00Z`

### 7.2 分页规范

| 参数 | 默认值 | 最大值 | 说明 |
|------|--------|--------|------|
| page | 1 | - | 页码，从1开始 |
| pageSize | 12 | 50 | 每页条数 |

### 7.3 文件上传规范

| 项目 | 规格 |
|------|------|
| 上传方式 | multipart/form-data |
| 音频格式 | MP3, WAV, M4A, OGG |
| 音频大小 | ≤50MB |
| 剧本格式 | txt (UTF-8/GBK) |
| 剧本大小 | ≤5MB |
| 上传超时 | 60s |
